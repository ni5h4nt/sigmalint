# RED001 — Detection fingerprint near-duplicates an existing SigmaHQ rule

## What it checks
Computes a **semantic fingerprint** over `(logsource, detection)` and
compares with the vendored SigmaHQ corpus. Reports matches with
Jaccard >= 0.85.

## Semantic canonicalization — "what counts as a duplicate"

A duplicate is not a textual duplicate; it is a duplicate of detection
intent. The fingerprint is constructed by:

1. **Selector names are dropped.** Two rules that name the same field
   pattern `selection` vs `proc_filter` are still semantically the
   same rule. Selector identity adds no detection signal.
2. **Field modifiers are normalized.** `|endswith`, `|contains`, and
   `|startswith` are mapped to a small canonical set so that
   functionally equivalent matches collapse together.
3. **AND/OR structure is preserved.** A rule that ORs three
   alternatives is distinct from a rule that ANDs three predicates,
   even if the leaves are identical. Detection semantics live in
   the boolean structure, not just the leaves.
4. **Logsource is included as-is.** A rule on `process_creation` is
   not a duplicate of an otherwise-identical rule on `network_connection`.

The Jaccard threshold of `0.85` is a **v0.1 calibration point**, not a
defended optimum. It was chosen by inspection on the SigmaHQ corpus
to surface clear duplicates without flooding on near-misses; future
work should tune this threshold empirically against a labeled corpus
of "true duplicate vs intentional variant" pairs.

## Why
Duplicating an upstream rule wastes review effort and increases
maintenance cost.

## Bad example
A locally authored rule that mirrors `proc_creation_win_*.yml` from
SigmaHQ with cosmetic changes only.

## Good example
A rule that adds a distinct selector, a different logsource, or a new
filter on top of the upstream baseline.

## How to fix
Either contribute back to SigmaHQ, or document the intentional
divergence and disable RED001 for that file.
