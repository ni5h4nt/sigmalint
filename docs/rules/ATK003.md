# ATK003 — logsource is plausible for cited techniques

## What it checks
Heuristic: the rule's `logsource.product`/`category` is consistent with
the platforms ATT&CK lists for the cited techniques.

## Why
A rule tagged with a Linux-only technique but tied to a Windows
process-creation logsource is almost certainly mistagged.

## Bad example
```yaml
logsource: { product: windows, category: process_creation }
tags: [attack.t1053.003]   # cron — linux only
```

## Good example
```yaml
logsource: { product: linux, category: process_creation }
tags: [attack.t1053.003]
```

## How to fix
Either correct the tag or correct the logsource. This is a weak
signal — review by hand.

## Caveats

The lookup table that drives this heuristic
(`src/sigmalint/data/vendored/attack-logsource-map.yml`) is
**sigmalint's own mapping** between Sigma logsource fields and ATT&CK
platforms — it is not a canonical MITRE artifact. The mapping reflects
the maintainers' best reading of which `(product, category)` tuples
are plausible for which ATT&CK platform values; expect drift if the
upstream Sigma `logsources/` taxonomy or MITRE's platform list
changes.

Further, ATT&CK technique platform metadata is not stable: MITRE
revises which platforms a technique applies to between ATT&CK
releases. This rule fires `info` (not `warning`) precisely because a
platform mismatch under one ATT&CK version may resolve in the next.
