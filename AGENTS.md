**15044** 3:43p 🔵 **Sigmalint v0.1 Design Specification Reviewed**
The sigmalint design specification defines a comprehensive Sigma detection rule linter. The architecture follows strict layering: core/ (rule framework, runner, scoring), data/ (reference data loaders), rules/ (one module per dimension), reporting/ (formatters), and cli/ (Typer entry point). The tool implements a two-layer validation model: SCHEMA rules gate validity (YAML parsing, JSON schema validation, required keys, condition grammar), and quality rules assess six dimensions (MITRE ATT&CK alignment, taxonomy field correctness, false-positive risk via four heuristics, metadata completeness, redundancy with SigmaHQ corpus, and Sigma style). Scoring applies severity-based weights (error=10, warning=3, info=1) per finding, computing dimension scores as max(0, 100 - sum of penalties), then a weighted total across dimensions. Three profiles tune severity: strict (maximum enforcement), sigmahq (matches submission expectations), and local (internal corpus). The design includes vendored data with cache-refreshable overrides via `sigmalint update-data`, plugin-shaped rule registry with decorator-based registration, and output in text/json/sarif/github formats. Testing strategy includes per-rule fixtures, golden-file integration tests, property-based condition parser tests, and 90% coverage gate. The three-week schedule targets core framework and 14 rules in week 1, remaining 12 rules and data loaders in week 2, and documentation/release/OSS readiness in week 3.
